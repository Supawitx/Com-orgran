## วัตถุประสงค์การเรียนรู้

หลังจากศึกษาบทนี้แล้ว ผู้เรียนควรสามารถ:

- สรุปหน้าที่สำคัญของระบบปฏิบัติการ (OS) ในระดับภาพรวมได้
- อภิปรายพัฒนาการของระบบปฏิบัติการตั้งแต่ระบบแบทช์อย่างง่ายในยุคแรกจนถึงระบบที่ซับซ้อนในปัจจุบันได้
- อธิบายความแตกต่างระหว่างการจัดตารางเวลาระยะยาว ระยะกลาง และระยะสั้นได้
- เข้าใจเหตุผลของการแบ่งพาร์ทิชันหน่วยความจำ และอธิบายเทคนิคต่าง ๆ ที่ใช้ได้
- ประเมินข้อได้เปรียบและข้อเสียเปรียบเชิงเปรียบเทียบระหว่างการแบ่งหน้าและการแบ่งเซกเมนต์ได้
- นิยามหน่วยความจำเสมือนได้

---

แม้ว่าจุดมุ่งเน้นของหนังสือเล่มนี้คือฮาร์ดแวร์คอมพิวเตอร์ แต่มีด้านหนึ่งของซอฟต์แวร์ที่จำเป็นต้องกล่าวถึง นั่นคือระบบปฏิบัติการ (OS) ของคอมพิวเตอร์ OS คือโปรแกรมที่จัดการทรัพยากรของคอมพิวเตอร์ ให้บริการแก่โปรแกรมเมอร์ และจัดตารางการประมวลผลของโปรแกรมอื่น ๆ การมีความเข้าใจเกี่ยวกับระบบปฏิบัติการเป็นสิ่งจำเป็นเพื่อให้เข้าใจกลไกที่ CPU ใช้ควบคุมระบบคอมพิวเตอร์ โดยเฉพาะอย่างยิ่ง คำอธิบายเกี่ยวกับผลของอินเทอร์รัปต์ (interrupt) และการจัดการลำดับชั้นหน่วยความจำ (memory hierarchy) จะอธิบายได้ดีที่สุดในบริบทนี้

บทนี้เริ่มต้นด้วยภาพรวมและประวัติโดยย่อของระบบปฏิบัติการ เนื้อหาหลักของบทนี้เน้นที่หน้าที่สองประการของ OS ที่มีความเกี่ยวข้องมากที่สุดกับการศึกษาโครงสร้างและสถาปัตยกรรมคอมพิวเตอร์ ได้แก่ การจัดตารางเวลา (scheduling) และการจัดการหน่วยความจำ (memory management)

## 8.1 ภาพรวมของระบบปฏิบัติการ

### วัตถุประสงค์และหน้าที่ของระบบปฏิบัติการ

OS คือโปรแกรมที่ควบคุมการประมวลผลของโปรแกรมประยุกต์ และทำหน้าที่เป็นส่วนติดต่อ (interface) ระหว่างโปรแกรมประยุกต์กับฮาร์ดแวร์คอมพิวเตอร์ อาจกล่าวได้ว่า OS มีวัตถุประสงค์สองประการ คือ:

- **ความสะดวกในการใช้งาน (Convenience):** OS ทำให้การใช้คอมพิวเตอร์มีความสะดวกมากขึ้น
- **ประสิทธิภาพ (Efficiency):** OS ช่วยให้ใช้ทรัพยากรของระบบคอมพิวเตอร์ได้อย่างมีประสิทธิภาพ

#### ระบบปฏิบัติการในฐานะส่วนติดต่อระหว่างผู้ใช้กับคอมพิวเตอร์ (The Operating System as a User/Computer Interface)

ฮาร์ดแวร์และซอฟต์แวร์ที่ใช้สำหรับให้บริการโปรแกรมประยุกต์แก่ผู้ใช้สามารถมองในลักษณะที่เป็นลำดับชั้น (layered) หรือแบบลำดับชั้นสูงต่ำ (hierarchical) ดังที่แสดงในรูปที่ 8.1 ผู้ใช้ปลายทาง (end user) โดยทั่วไปไม่สนใจสถาปัตยกรรมของคอมพิวเตอร์ แต่มองระบบคอมพิวเตอร์ในแง่ของโปรแกรมประยุกต์ โปรแกรมประยุกต์นั้นอาจเขียนด้วยภาษาโปรแกรมและพัฒนาโดยโปรแกรมเมอร์ประยุกต์

การพัฒนาโปรแกรมประยุกต์ที่รับผิดชอบการควบคุมฮาร์ดแวร์คอมพิวเตอร์ทั้งหมดโดยตรงถือเป็นงานที่ซับซ้อนอย่างยิ่ง เพื่อบรรเทาความซับซ้อนนี้ จึงมีการจัดเตรียมโปรแกรมระบบ (system programs) ชุดหนึ่ง บางโปรแกรมเรียกว่า **ยูทิลิตี (utilities)** ซึ่งทำหน้าที่ที่ใช้บ่อยในการช่วยสร้างโปรแกรม จัดการไฟล์ และควบคุมอุปกรณ์ I/O โปรแกรมระบบที่สำคัญที่สุดคือ OS ซึ่งซ่อนรายละเอียดของฮาร์ดแวร์จากโปรแกรมเมอร์และให้ส่วนติดต่อที่สะดวกในการใช้ระบบ ทำหน้าที่เป็นตัวกลางเพื่ออำนวยความสะดวกแก่โปรแกรมเมอร์และโปรแกรมประยุกต์ในการเข้าถึงและใช้งานสิ่งอำนวยความสะดวกและบริการต่าง ๆ

โดยสังเขป OS มักให้บริการในด้านต่าง ๆ ดังต่อไปนี้:

- **การสร้างโปรแกรม (Program creation):** OS จัดเตรียมสิ่งอำนวยความสะดวกและบริการหลากหลาย เช่น โปรแกรมแก้ไขข้อความ (editor) และโปรแกรมค้นหาข้อผิดพลาด (debugger) เพื่อช่วยโปรแกรมเมอร์ในการสร้างโปรแกรม โดยทั่วไปบริการเหล่านี้อยู่ในรูปของโปรแกรมยูทิลิตีที่ไม่ได้เป็นส่วนหนึ่งของ OS โดยตรง แต่เข้าถึงได้ผ่าน OS
- **การประมวลผลโปรแกรม (Program execution):** มีขั้นตอนจำนวนหนึ่งที่ต้องดำเนินการเพื่อประมวลผลโปรแกรม ได้แก่ การโหลดคำสั่งและข้อมูลลงในหน่วยความจำหลัก การเตรียมความพร้อมของอุปกรณ์ I/O และไฟล์ รวมถึงการเตรียมทรัพยากรอื่น ๆ OS จัดการสิ่งเหล่านี้ทั้งหมดให้แก่ผู้ใช้
- **การเข้าถึงอุปกรณ์ I/O (Access to I/O devices):** อุปกรณ์ I/O แต่ละชิ้นต้องใช้ชุดคำสั่งหรือสัญญาณควบคุมเฉพาะของตน OS จัดการรายละเอียดเหล่านี้เพื่อให้โปรแกรมเมอร์คิดเพียงในแง่ของการอ่านและการเขียนอย่างง่าย
- **การควบคุมการเข้าถึงไฟล์ (Controlled access to files):** ในกรณีของไฟล์ การควบคุมต้องอาศัยความเข้าใจทั้งลักษณะของอุปกรณ์ I/O (ฮาร์ดดิสก์ เทป) และรูปแบบไฟล์บนสื่อเก็บข้อมูล OS จัดการรายละเอียดเหล่านี้ นอกจากนี้ ในระบบที่มีผู้ใช้หลายคนพร้อมกัน OS ยังให้กลไกการป้องกันเพื่อควบคุมการเข้าถึงไฟล์
- **การเข้าถึงระบบ (System access):** ในกรณีของระบบที่ใช้ร่วมกันหรือระบบสาธารณะ OS ควบคุมการเข้าถึงทั้งระบบโดยรวมและทรัพยากรระบบเฉพาะเจาะจง ฟังก์ชันการเข้าถึงต้องให้การป้องกันทรัพยากรและข้อมูลจากผู้ใช้ที่ไม่ได้รับอนุญาต และต้องแก้ไขข้อขัดแย้งในการแย่งชิงทรัพยากร
- **การตรวจจับและตอบสนองต่อข้อผิดพลาด (Error detection and response):** ข้อผิดพลาดหลากหลายประเภทอาจเกิดขึ้นขณะที่ระบบคอมพิวเตอร์ทำงาน ซึ่งรวมถึงข้อผิดพลาดของฮาร์ดแวร์ทั้งภายในและภายนอก เช่น ข้อผิดพลาดของหน่วยความจำหรืออุปกรณ์เสียหาย รวมถึงข้อผิดพลาดของซอฟต์แวร์ต่าง ๆ เช่น เลขคณิตล้นค่า (arithmetic overflow) การพยายามเข้าถึงตำแหน่งหน่วยความจำที่ไม่ได้รับอนุญาต และความไม่สามารถของ OS ในการตอบสนองคำขอของโปรแกรมประยุกต์ ในทุกกรณี OS ต้องตอบสนองเพื่อแก้ไขสภาพข้อผิดพลาดโดยส่งผลกระทบต่อการทำงานของโปรแกรมที่รันอยู่น้อยที่สุด การตอบสนองอาจมีตั้งแต่การยุติโปรแกรมที่ก่อให้เกิดข้อผิดพลาด การลองดำเนินการใหม่ หรือเพียงแค่รายงานข้อผิดพลาดให้กับโปรแกรมประยุกต์
- **การบัญชีทรัพยากร (Accounting):** OS ที่ดีจะเก็บสถิติการใช้งานสำหรับทรัพยากรต่าง ๆ และตรวจสอบพารามิเตอร์ประสิทธิภาพ เช่น เวลาตอบสนอง ในระบบใดก็ตาม ข้อมูลนี้มีประโยชน์ในการคาดการณ์ความต้องการปรับปรุงในอนาคตและปรับแต่งระบบเพื่อปรับปรุงประสิทธิภาพ ในระบบที่มีผู้ใช้หลายคน ข้อมูลนี้สามารถนำไปใช้เพื่อวัตถุประสงค์ในการเรียกเก็บค่าบริการ

รูปที่ 8.1 ยังแสดงส่วนติดต่อหลักสามส่วนในระบบคอมพิวเตอร์ทั่วไป:

- **สถาปัตยกรรมชุดคำสั่ง (Instruction Set Architecture — ISA):** ISA กำหนดรายการคำสั่งภาษาเครื่อง (machine language instructions) ที่คอมพิวเตอร์สามารถปฏิบัติตามได้ ส่วนติดต่อนี้เป็นเส้นแบ่งระหว่างฮาร์ดแวร์และซอฟต์แวร์ ทั้งโปรแกรมประยุกต์และยูทิลิตีอาจเข้าถึง ISA ได้โดยตรง สำหรับโปรแกรมเหล่านี้จะมีชุดย่อยของรายการคำสั่งที่ใช้งานได้ (user ISA) ส่วน OS มีสิทธิ์เข้าถึงคำสั่งภาษาเครื่องเพิ่มเติมที่เกี่ยวข้องกับการจัดการทรัพยากรระบบ (system ISA)
- **ส่วนติดต่อแบบไบนารีของโปรแกรมประยุกต์ (Application Binary Interface — ABI):** ABI กำหนดมาตรฐานสำหรับความสามารถในการพกพาของไบนารีระหว่างโปรแกรม ABI กำหนดส่วนติดต่อการเรียกระบบ (system call interface) ไปยังระบบปฏิบัติการ และทรัพยากรฮาร์ดแวร์และบริการที่ใช้งานได้ในระบบผ่าน user ISA
- **ส่วนติดต่อการโปรแกรมประยุกต์ (Application Programming Interface — API):** API ให้โปรแกรมสามารถเข้าถึงทรัพยากรฮาร์ดแวร์และบริการที่ใช้งานได้ในระบบผ่าน user ISA ที่เสริมด้วยการเรียกไลบรารีภาษาระดับสูง (High-Level Language — HLL) การเรียกระบบใด ๆ มักดำเนินการผ่านไลบรารี การใช้ API ช่วยให้ซอฟต์แวร์ประยุกต์สามารถพอร์ต (port) ไปยังระบบอื่นที่รองรับ API เดียวกันได้ง่ายผ่านการคอมไพล์ใหม่

#### ระบบปฏิบัติการในฐานะผู้จัดการทรัพยากร (The Operating System as Resource Manager)

คอมพิวเตอร์คือชุดของทรัพยากรสำหรับการเคลื่อนย้าย เก็บ และประมวลผลข้อมูล และสำหรับการควบคุมฟังก์ชันเหล่านี้ OS มีหน้าที่จัดการทรัพยากรเหล่านี้

OS มีลักษณะพิเศษในฐานะกลไกควบคุมสองประการ คือ:

1. OS ทำงานในลักษณะเดียวกับซอฟต์แวร์คอมพิวเตอร์ทั่วไป กล่าวคือมันเป็นโปรแกรมที่ประมวลผลโดยโปรเซสเซอร์
2. OS มักสละการควบคุมและต้องพึ่งพาโปรเซสเซอร์เพื่อให้ได้รับการควบคุมกลับคืน

OS ควบคุมโปรเซสเซอร์ในการใช้ทรัพยากรระบบอื่น ๆ และในการกำหนดเวลาการประมวลผลของโปรแกรมอื่น ๆ แต่เพื่อให้โปรเซสเซอร์ทำสิ่งเหล่านี้ได้ มันต้องหยุดการประมวลผลโปรแกรม OS และประมวลผลโปรแกรมอื่น ดังนั้น OS จึงสละการควบคุมเพื่อให้โปรเซสเซอร์ทำงาน "ที่มีประโยชน์" แล้วจึงกลับมาควบคุมอีกครั้งเพื่อเตรียมโปรเซสเซอร์สำหรับงานถัดไป

ส่วนหนึ่งของ OS อยู่ในหน่วยความจำหลัก ซึ่งรวมถึง **เคอร์เนล (kernel)** หรือ **นิวเคลียส (nucleus)** ที่บรรจุฟังก์ชันที่ใช้บ่อยที่สุดใน OS OS ตัดสินใจว่าเมื่อใดอุปกรณ์ I/O จะถูกใช้โดยโปรแกรมที่กำลังประมวลผล และควบคุมการเข้าถึงและการใช้ไฟล์ โปรเซสเซอร์เองก็เป็นทรัพยากร และ OS ต้องกำหนดว่าจะใช้เวลาโปรเซสเซอร์เท่าไรสำหรับการประมวลผลโปรแกรมผู้ใช้หนึ่ง ๆ ในกรณีของระบบที่มีหลายโปรเซสเซอร์ การตัดสินใจนี้ต้องครอบคลุมโปรเซสเซอร์ทั้งหมด

### ประเภทของระบบปฏิบัติการ

ลักษณะสำคัญบางประการช่วยแยกความแตกต่างของระบบปฏิบัติการประเภทต่าง ๆ ลักษณะเหล่านี้แบ่งตามมิติอิสระสองมิติ

มิติแรกระบุว่าระบบเป็นแบบแบทช์ (batch) หรือแบบโต้ตอบ (interactive) ในระบบแบบโต้ตอบ ผู้ใช้/โปรแกรมเมอร์โต้ตอบกับคอมพิวเตอร์โดยตรง โดยปกติผ่านคีย์บอร์ด/จอแสดงผล เพื่อขอให้ประมวลผลงาน (job) หรือดำเนินการทรานแซกชัน นอกจากนี้ ผู้ใช้อาจสื่อสารกับคอมพิวเตอร์ในระหว่างการประมวลผลงานด้วย ขึ้นอยู่กับลักษณะของโปรแกรมประยุกต์ **ระบบแบทช์ (batch system)** เป็นสิ่งตรงข้ามกับระบบแบบโต้ตอบ โปรแกรมของผู้ใช้จะถูกรวมเข้ากับโปรแกรมของผู้ใช้คนอื่น ๆ และส่งโดยผู้ดำเนินการคอมพิวเตอร์ หลังจากโปรแกรมเสร็จสิ้น ผลลัพธ์จะถูกพิมพ์ออกมาให้ผู้ใช้ ระบบแบทช์แบบบริสุทธิ์นั้นหายากในปัจจุบัน

มิติอิสระอีกมิติหนึ่งระบุว่าระบบใช้ **การทำงานหลายโปรแกรมพร้อมกัน (multiprogramming)** หรือไม่ ในการทำงานแบบ multiprogramming จะพยายามรักษาโปรเซสเซอร์ให้ทำงานอยู่ตลอดเวลาโดยให้ทำงานกับโปรแกรมมากกว่าหนึ่งโปรแกรมในเวลาเดียวกัน โปรแกรมหลายโปรแกรมจะถูกโหลดเข้าหน่วยความจำ และโปรเซสเซอร์จะสลับระหว่างโปรแกรมเหล่านั้นอย่างรวดเร็ว ทางเลือกอีกทางคือระบบ **การทำงานโปรแกรมเดียว (uniprogramming)** ที่ทำงานกับโปรแกรมเดียวในแต่ละครั้ง

#### ระบบยุคแรก (Early Systems)

ในช่วงคอมพิวเตอร์ยุคแรก ตั้งแต่ปลายทศวรรษ 1940 ถึงกลางทศวรรษ 1950 โปรแกรมเมอร์โต้ตอบกับฮาร์ดแวร์คอมพิวเตอร์โดยตรงโดยไม่มี OS คอมพิวเตอร์ในยุคนั้นถูกควบคุมจากคอนโซล (console) ซึ่งประกอบด้วยไฟแสดงสถานะ สวิตช์สลับ อุปกรณ์รับข้อมูลบางรูปแบบ และเครื่องพิมพ์ โปรแกรมในรหัสโปรเซสเซอร์ถูกโหลดผ่านอุปกรณ์รับข้อมูล (เช่น เครื่องอ่านบัตร) หากข้อผิดพลาดทำให้โปรแกรมหยุด สภาพข้อผิดพลาดจะแสดงผ่านไฟ โปรแกรมเมอร์สามารถตรวจสอบรีจิสเตอร์และหน่วยความจำหลักเพื่อหาสาเหตุของข้อผิดพลาดได้

ระบบยุคแรกเหล่านี้มีปัญหาหลักสองประการ:

- **การจัดตารางเวลา (Scheduling):** การติดตั้งส่วนใหญ่ใช้ใบลงชื่อเพื่อจองเวลาโปรเซสเซอร์ ผู้ใช้อาจลงชื่อขอใช้หนึ่งชั่วโมงแต่เสร็จใน 45 นาที ส่งผลให้เวลาคอมพิวเตอร์ว่างสูญเปล่า หรืออาจพบปัญหาและไม่สามารถเสร็จภายในเวลาที่จัดสรร ทำให้ต้องหยุดก่อนที่จะแก้ปัญหาได้
- **เวลาการตั้งค่า (Setup time):** โปรแกรมเดียว ซึ่งเรียกว่า **งาน (job)** อาจเกี่ยวข้องกับการโหลดคอมไพเลอร์และโปรแกรมในภาษาระดับสูง (source program) เข้าหน่วยความจำ การบันทึกโปรแกรมที่คอมไพล์แล้ว (object program) และการโหลดและเชื่อมโยงโปรแกรม object กับฟังก์ชันทั่วไป แต่ละขั้นตอนอาจต้องติดตั้งหรือถอดเทป หรือจัดชุดบัตร หากเกิดข้อผิดพลาด ผู้ใช้มักต้องกลับไปเริ่มต้นขั้นตอนการตั้งค่าใหม่ทั้งหมด

โหมดการทำงานนี้อาจเรียกว่า **การประมวลผลแบบต่อเนื่อง (serial processing)** สะท้อนความจริงที่ว่าผู้ใช้มีสิทธิ์เข้าถึงคอมพิวเตอร์ตามลำดับ

#### ระบบแบทช์อย่างง่าย (Simple Batch Systems)

โปรเซสเซอร์ยุคแรกมีราคาแพงมาก จึงจำเป็นต้องเพิ่มประสิทธิภาพการใช้โปรเซสเซอร์ให้สูงสุด เพื่อปรับปรุงการใช้งาน จึงมีการพัฒนาระบบปฏิบัติการแบบแบทช์อย่างง่าย

ในระบบเช่นนี้ ซึ่งเรียกอีกอย่างว่า **มอนิเตอร์ (monitor)** ผู้ใช้ไม่มีการเข้าถึงโปรเซสเซอร์โดยตรงอีกต่อไป ผู้ใช้ส่งงานบนบัตรหรือเทปให้กับผู้ดำเนินการคอมพิวเตอร์ ซึ่งรวบรวมงานเหล่านั้นตามลำดับและวางทั้งชุดบนอุปกรณ์รับข้อมูล เพื่อให้มอนิเตอร์ใช้งาน

ส่วนของมอนิเตอร์ที่ต้องอยู่ในหน่วยความจำหลักตลอดเวลาเรียกว่า **มอนิเตอร์ประจำ (resident monitor)** มอนิเตอร์อ่านงานทีละงานจากอุปกรณ์รับข้อมูล เมื่องานถูกอ่านเข้ามา งานปัจจุบันจะถูกวางในพื้นที่โปรแกรมของผู้ใช้และมอบการควบคุมให้กับงานนั้น เมื่องานเสร็จสิ้น มันจะคืนการควบคุมให้กับมอนิเตอร์ ซึ่งจะอ่านงานถัดไปทันที

คุณลักษณะฮาร์ดแวร์บางอย่างที่เป็นประโยชน์สำหรับระบบแบทช์อย่างง่าย:

- **การป้องกันหน่วยความจำ (Memory protection):** ขณะที่โปรแกรมของผู้ใช้กำลังทำงาน มันต้องไม่เปลี่ยนแปลงพื้นที่หน่วยความจำที่บรรจุมอนิเตอร์ หากมีความพยายามเช่นนั้น ฮาร์ดแวร์โปรเซสเซอร์ควรตรวจพบข้อผิดพลาดและส่งการควบคุมไปยังมอนิเตอร์
- **ตัวจับเวลา (Timer):** ตัวจับเวลาใช้เพื่อป้องกันไม่ให้งานเดียวครอบครองระบบ ตัวจับเวลาถูกตั้งไว้ที่จุดเริ่มต้นของแต่ละงาน หากตัวจับเวลาหมดเวลา จะเกิดอินเทอร์รัปต์และการควบคุมจะกลับไปยังมอนิเตอร์
- **คำสั่งสิทธิพิเศษ (Privileged instructions):** คำสั่งบางอย่างถูกกำหนดให้มีสิทธิพิเศษและสามารถประมวลผลได้โดยมอนิเตอร์เท่านั้น หากโปรเซสเซอร์พบคำสั่งดังกล่าวขณะประมวลผลโปรแกรมของผู้ใช้ จะเกิดอินเทอร์รัปต์ข้อผิดพลาด ในบรรดาคำสั่งสิทธิพิเศษ ได้แก่ คำสั่ง I/O เพื่อให้มอนิเตอร์รักษาการควบคุมอุปกรณ์ I/O ทั้งหมด
- **อินเทอร์รัปต์ (Interrupts):** คุณลักษณะนี้ให้ความยืดหยุ่นแก่ OS มากขึ้นในการสละการควบคุมและกลับมาควบคุมจากโปรแกรมผู้ใช้อีกครั้ง

#### ระบบแบทช์แบบหลายโปรแกรม (Multiprogrammed Batch Systems)

แม้แต่กับการจัดลำดับงานอัตโนมัติที่ระบบแบทช์อย่างง่ายให้ โปรเซสเซอร์ก็มักไม่ทำงาน ปัญหาคืออุปกรณ์ I/O ช้ากว่าโปรเซสเซอร์มาก ตัวอย่างในรูปที่ 8.4 แสดงการคำนวณสำหรับโปรแกรมที่ประมวลผลไฟล์บันทึกและดำเนินการ 100 คำสั่งโปรเซสเซอร์โดยเฉลี่ยต่อบันทึก ในตัวอย่างนี้ คอมพิวเตอร์ใช้เวลากว่า 96% ในการรออุปกรณ์ I/O ให้ถ่ายโอนข้อมูลเสร็จสิ้น

```
อ่านหนึ่งบันทึกจากไฟล์    15 ไมโครวินาที
ประมวลผล 100 คำสั่ง        1 ไมโครวินาที
เขียนหนึ่งบันทึกลงไฟล์    15 ไมโครวินาที
รวม                         31 ไมโครวินาที

เปอร์เซ็นต์การใช้งาน CPU = 1/31 = 0.032 = 3.2%
```

หากมีพื้นที่สำหรับ OS และโปรแกรมผู้ใช้สองโปรแกรม เมื่อโปรแกรมหนึ่งต้องรอ I/O โปรเซสเซอร์สามารถสลับไปยังโปรแกรมอื่นได้ เทคนิคนี้เรียกว่า **การทำงานหลายโปรแกรมพร้อมกัน (multiprogramming)** หรือ **มัลติทาสกิ้ง (multitasking)** ซึ่งเป็นแนวคิดหลักของระบบปฏิบัติการสมัยใหม่

**ตัวอย่างที่ 8.1** แสดงให้เห็นถึงประโยชน์ของการทำงานหลายโปรแกรม โดยพิจารณาคอมพิวเตอร์ที่มีหน่วยความจำว่าง 250 เมกาไบต์ มีดิสก์ เทอร์มินัล และเครื่องพิมพ์ โปรแกรมสามโปรแกรมคือ JOB1, JOB2 และ JOB3 ถูกส่งเพื่อประมวลผลพร้อมกัน ผลของการทำงานหลายโปรแกรมเทียบกับการทำงานโปรแกรมเดียวแสดงในตารางที่ 8.2:

| ตัวชี้วัด | การทำงานโปรแกรมเดียว | การทำงานหลายโปรแกรม |
|---|---|---|
| การใช้งานโปรเซสเซอร์ (%) | 20 | 40 |
| การใช้งานหน่วยความจำ (%) | 33 | 67 |
| การใช้งานดิสก์ (%) | 33 | 67 |
| การใช้งานเครื่องพิมพ์ (%) | 33 | 67 |
| เวลาที่ผ่านไป (นาที) | 30 | 15 |
| อัตราปริมาณงาน (งาน/ชั่วโมง) | 6 | 12 |
| เวลาตอบสนองเฉลี่ย (นาที) | 18 | 10 |

#### ระบบแบ่งเวลา (Time-Sharing Systems)

เช่นเดียวกับที่การทำงานหลายโปรแกรมช่วยให้โปรเซสเซอร์จัดการงานแบทช์หลายงานในเวลาเดียวกัน การทำงานหลายโปรแกรมยังสามารถใช้เพื่อจัดการงานแบบโต้ตอบหลายงานได้ ในกรณีหลังนี้ เทคนิคนี้เรียกว่า **การแบ่งเวลา (time sharing)** เนื่องจากเวลาของโปรเซสเซอร์ถูกแบ่งปันในหมู่ผู้ใช้หลายคน

ในระบบแบ่งเวลา ผู้ใช้หลายคนเข้าถึงระบบพร้อมกันผ่านเทอร์มินัล โดย OS จะสลับการประมวลผลของแต่ละโปรแกรมผู้ใช้ในช่วงเวลาสั้น ๆ หรือ **ควอนตัม (quantum)** ของการคำนวณ ดังนั้น หากมีผู้ใช้ n คนที่กำลังขอรับบริการพร้อมกัน ผู้ใช้แต่ละคนจะเห็นเพียงความเร็วคอมพิวเตอร์เฉลี่ย 1/n เท่านั้น ไม่นับค่าใช้จ่าย (overhead) ของ OS

## 8.2 การจัดตารางเวลา (Scheduling)

กุญแจสำคัญของการทำงานหลายโปรแกรมคือการจัดตารางเวลา ซึ่งมีการจัดตารางเวลาสี่ประเภทที่เกี่ยวข้องโดยทั่วไป ดังแสดงในตารางที่ 8.4

ก่อนอื่น ขอแนะนำแนวคิดของ **กระบวนการ (process)** คำนี้ถูกใช้ครั้งแรกโดยผู้ออกแบบ OS Multics ในทศวรรษ 1960 เป็นคำที่มีความหมายกว้างกว่าคำว่า *งาน (job)* คำนิยามของ process มีหลายแบบ รวมถึง:

- โปรแกรมที่กำลังประมวลผล
- "จิตวิญญาณที่มีชีวิต" ของโปรแกรม
- สิ่งที่โปรเซสเซอร์ถูกกำหนดให้ทำงาน

| ประเภทการจัดตารางเวลา | คำอธิบาย |
|---|---|
| การจัดตารางเวลาระยะยาว (Long-term scheduling) | การตัดสินใจเพิ่มกระบวนการเข้าสู่กลุ่มกระบวนการที่จะประมวลผล |
| การจัดตารางเวลาระยะกลาง (Medium-term scheduling) | การตัดสินใจเพิ่มจำนวนกระบวนการที่อยู่ในหน่วยความจำหลักบางส่วนหรือทั้งหมด |
| การจัดตารางเวลาระยะสั้น (Short-term scheduling) | การตัดสินใจว่ากระบวนการที่พร้อมใช้งานใดจะถูกประมวลผลโดยโปรเซสเซอร์ |
| การจัดตารางเวลา I/O (I/O scheduling) | การตัดสินใจว่าคำขอ I/O ที่รอดำเนินการของกระบวนการใดจะถูกจัดการโดยอุปกรณ์ I/O ที่ว่างอยู่ |

### การจัดตารางเวลาระยะยาว (Long-Term Scheduling)

ตัวจัดตารางเวลาระยะยาว (long-term scheduler) กำหนดว่าโปรแกรมใดได้รับการรับเข้าสู่ระบบเพื่อประมวลผล ดังนั้น มันควบคุมระดับของการทำงานหลายโปรแกรม (จำนวนกระบวนการในหน่วยความจำ) เมื่อได้รับการรับเข้า งานหรือโปรแกรมผู้ใช้จะกลายเป็นกระบวนการและถูกเพิ่มในคิวสำหรับตัวจัดตารางเวลาระยะสั้น

ในระบบแบทช์ งานที่ส่งมาใหม่จะถูกส่งไปที่ดิสก์และเก็บไว้ในคิวแบทช์ ตัวจัดตารางเวลาระยะยาวสร้างกระบวนการจากคิวเมื่อทำได้ มีการตัดสินใจสองอย่างที่เกี่ยวข้อง ได้แก่ ตัวจัดตารางต้องตัดสินใจว่า OS สามารถรับกระบวนการเพิ่มเติมได้หรือไม่ และตัดสินใจว่าจะยอมรับงานใดเพื่อเปลี่ยนเป็นกระบวนการ เกณฑ์ที่ใช้อาจรวมถึงลำดับความสำคัญ (priority) เวลาประมวลผลที่คาดหวัง และความต้องการ I/O

### การจัดตารางเวลาระยะกลาง (Medium-Term Scheduling)

การจัดตารางเวลาระยะกลางเป็นส่วนหนึ่งของฟังก์ชันการสลับ (swapping) โดยทั่วไป การตัดสินใจสลับเข้า (swapping-in) ขึ้นอยู่กับความจำเป็นในการจัดการระดับของการทำงานหลายโปรแกรม ในระบบที่ไม่ใช้หน่วยความจำเสมือน การจัดการหน่วยความจำก็เป็นประเด็นสำคัญด้วย

### การจัดตารางเวลาระยะสั้น (Short-Term Scheduling)

ตัวจัดตารางเวลาระยะยาวทำงานค่อนข้างน้อยครั้งและตัดสินใจในระดับหยาบ ส่วนตัวจัดตารางเวลาระยะสั้น หรือที่เรียกว่า **ดิสแพตเชอร์ (dispatcher)** ทำงานบ่อยครั้งและตัดสินใจในระดับละเอียดว่างานใดจะถูกประมวลผลถัดไป

#### สถานะของกระบวนการ (Process States)

เพื่อเข้าใจการทำงานของตัวจัดตารางเวลาระยะสั้น จำเป็นต้องพิจารณาแนวคิดของ **สถานะกระบวนการ (process state)** ตลอดช่วงชีวิตของกระบวนการ สถานะของมันจะเปลี่ยนแปลงหลายครั้ง อย่างน้อยมีสถานะที่กำหนดไว้ห้าสถานะสำหรับกระบวนการ:

- **ใหม่ (New):** โปรแกรมได้รับการรับเข้าโดยตัวจัดตารางระดับสูง แต่ยังไม่พร้อมที่จะประมวลผล OS จะเริ่มต้นกระบวนการและย้ายมันไปยังสถานะพร้อม
- **พร้อม (Ready):** กระบวนการพร้อมที่จะประมวลผลและกำลังรอการเข้าถึงโปรเซสเซอร์
- **กำลังทำงาน (Running):** กระบวนการกำลังถูกประมวลผลโดยโปรเซสเซอร์
- **รอ (Waiting):** กระบวนการถูกระงับการประมวลผลเนื่องจากรอทรัพยากรระบบบางอย่าง เช่น I/O
- **หยุด (Halted):** กระบวนการสิ้นสุดลงและจะถูกทำลายโดย OS

สำหรับแต่ละกระบวนการในระบบ OS ต้องรักษาข้อมูลที่แสดงสถานะของกระบวนการและข้อมูลอื่น ๆ ที่จำเป็นสำหรับการประมวลผลกระบวนการ เพื่อจุดประสงค์นี้ แต่ละกระบวนการจะถูกแทนด้วย **บล็อกควบคุมกระบวนการ (Process Control Block — PCB)** ซึ่งโดยทั่วไปประกอบด้วย:

- **ตัวระบุ (Identifier):** กระบวนการปัจจุบันแต่ละกระบวนการมีตัวระบุเฉพาะ
- **สถานะ (State):** สถานะปัจจุบันของกระบวนการ (ใหม่ พร้อม เป็นต้น)
- **ลำดับความสำคัญ (Priority):** ระดับลำดับความสำคัญสัมพัทธ์
- **ตัวนับโปรแกรม (Program counter):** ที่อยู่ของคำสั่งถัดไปในโปรแกรมที่จะประมวลผล
- **ตัวชี้หน่วยความจำ (Memory pointers):** ตำแหน่งเริ่มต้นและสิ้นสุดของกระบวนการในหน่วยความจำ
- **ข้อมูลบริบท (Context data):** ข้อมูลที่อยู่ในรีจิสเตอร์ของโปรเซสเซอร์ขณะที่กระบวนการกำลังทำงาน ข้อมูลบริบทและตัวนับโปรแกรมจะถูกบันทึกไว้เมื่อกระบวนการออกจากสถานะกำลังทำงาน และจะถูกดึงกลับโดยโปรเซสเซอร์เมื่อกลับมาประมวลผลกระบวนการ
- **ข้อมูลสถานะ I/O (I/O status information):** รวมถึงคำขอ I/O ที่ค้างอยู่ อุปกรณ์ I/O ที่ถูกกำหนดให้กับกระบวนการนี้ รายการไฟล์ที่ถูกกำหนดให้กับกระบวนการ เป็นต้น
- **ข้อมูลการบัญชี (Accounting information):** อาจรวมถึงจำนวนเวลาโปรเซสเซอร์และเวลาทางนาฬิกาที่ใช้ ขีดจำกัดเวลา หมายเลขบัญชี เป็นต้น

#### เทคนิคการจัดตารางเวลา (Scheduling Techniques)

OS รักษาคิวจำนวนหนึ่ง แต่ละคิวเป็นรายการรอของกระบวนการที่รอทรัพยากรบางอย่าง **คิวระยะยาว (long-term queue)** เป็นรายการงานที่รอใช้ระบบ **คิวระยะสั้น (short-term queue)** ประกอบด้วยกระบวนการทั้งหมดในสถานะพร้อม โดยทั่วไปการเลือกจะทำด้วย **อัลกอริทึมแบบวนรอบ (round-robin algorithm)** โดยให้เวลาแต่ละกระบวนการบ้าง นอกจากนี้ยังอาจใช้ระดับลำดับความสำคัญด้วย และมี **คิว I/O** สำหรับอุปกรณ์ I/O แต่ละชนิด

## 8.3 การจัดการหน่วยความจำ (Memory Management)

ในระบบที่ทำงานโปรแกรมเดียว หน่วยความจำหลักแบ่งออกเป็นสองส่วน ได้แก่ ส่วนสำหรับ OS (resident monitor) และส่วนสำหรับโปรแกรมที่กำลังประมวลผลอยู่ ในระบบที่ทำงานหลายโปรแกรม ส่วน "ผู้ใช้" ของหน่วยความจำจะถูกแบ่งย่อยเพื่อรองรับกระบวนการหลายกระบวนการ งานการแบ่งย่อยนี้ดำเนินการอย่างพลวัตโดย OS และเรียกว่า **การจัดการหน่วยความจำ (memory management)**

การจัดการหน่วยความจำที่มีประสิทธิผลเป็นสิ่งสำคัญยิ่งในระบบที่ทำงานหลายโปรแกรม หากมีเพียงไม่กี่กระบวนการในหน่วยความจำ กระบวนการทั้งหมดจะรอ I/O เป็นส่วนใหญ่ตลอดเวลา และโปรเซสเซอร์จะไม่ทำงาน ดังนั้น จำเป็นต้องจัดสรรหน่วยความจำอย่างมีประสิทธิภาพเพื่อบรรจุกระบวนการให้ได้มากที่สุดในหน่วยความจำ

### การสลับ (Swapping)

เมื่อกระบวนการทั้งหมดในหน่วยความจำไม่อยู่ในสถานะพร้อม (เช่น ทั้งหมดรออยู่กับการดำเนินการ I/O) แทนที่จะปล่อยให้ไม่ทำงาน โปรเซสเซอร์จะ **สลับ (swap)** กระบวนการหนึ่งออกไปยังดิสก์ไปยัง **คิวกลาง (intermediate queue)** นี่คือคิวของกระบวนการที่มีอยู่แล้วที่ถูกย้ายออกจากหน่วยความจำชั่วคราว

จากนั้น OS จะนำกระบวนการอื่นมาจากคิวกลาง หรืออนุมัติคำขอกระบวนการใหม่จากคิวระยะยาว แล้วดำเนินการต่อกับกระบวนการที่เพิ่งมาถึง

การสลับเป็นการดำเนินการ I/O จึงมีโอกาสที่จะทำให้ปัญหาแย่ลง ไม่ใช่ดีขึ้น แต่เนื่องจาก I/O ของดิสก์โดยทั่วไปเป็น I/O ที่เร็วที่สุดในระบบ การสลับมักจะช่วยปรับปรุงประสิทธิภาพ

### การแบ่งพาร์ทิชัน (Partitioning)

#### พาร์ทิชันขนาดคงที่ (Fixed-Size Partitions)

แผนการที่ง่ายที่สุดสำหรับการแบ่งพาร์ทิชันหน่วยความจำที่ว่างคือการใช้ **พาร์ทิชันขนาดคงที่ (fixed-size partitions)** แม้ว่าพาร์ทิชันจะมีขนาดคงที่ แต่ไม่จำเป็นต้องมีขนาดเท่ากัน เมื่อกระบวนการถูกนำเข้าหน่วยความจำ มันจะถูกวางไว้ในพาร์ทิชันที่เล็กที่สุดที่ยังว่างและสามารถบรรจุได้

แม้แต่การใช้พาร์ทิชันขนาดคงที่ที่ไม่เท่ากัน ก็ยังมีหน่วยความจำสูญเปล่า ตัวอย่างเช่น กระบวนการที่ต้องการหน่วยความจำ 3 เมกาไบต์จะถูกวางในพาร์ทิชัน 4 เมกาไบต์ ทำให้ 1 เมกาไบต์ที่อาจนำไปใช้กับกระบวนการอื่นถูกสูญเปล่า

#### พาร์ทิชันขนาดแปรผัน (Variable-Size Partitions)

แนวทางที่มีประสิทธิภาพกว่าคือการใช้ **พาร์ทิชันขนาดแปรผัน** เมื่อกระบวนการถูกนำเข้าหน่วยความจำ มันจะได้รับการจัดสรรหน่วยความจำพอดีกับที่ต้องการ ไม่มากไม่น้อย

อย่างไรก็ตาม วิธีนี้เริ่มต้นได้ดี แต่ในที่สุดนำไปสู่สถานการณ์ที่มีช่องว่างเล็กๆ จำนวนมากในหน่วยความจำ เมื่อเวลาผ่านไป หน่วยความจำจะกระจัดกระจายมากขึ้นและประสิทธิภาพการใช้งานลดลง เทคนิคเพื่อแก้ปัญหานี้เรียกว่า **การอัดแน่น (compaction):** OS จะย้ายกระบวนการในหน่วยความจำเป็นครั้งคราวเพื่อรวมหน่วยความจำว่างทั้งหมดไว้ในบล็อกเดียว แต่นี่เป็นกระบวนการที่ใช้เวลานานและสิ้นเปลืองเวลาโปรเซสเซอร์

#### ที่อยู่ตรรกะและทางกายภาพ (Logical and Physical Addresses)

เพื่อแก้ปัญหาที่กระบวนการไม่ได้ถูกโหลดไปยังตำแหน่งเดียวกันในหน่วยความจำหลักทุกครั้ง จึงมีการแยกความแตกต่างระหว่าง **ที่อยู่ตรรกะ (logical address)** และ **ที่อยู่ทางกายภาพ (physical address)**

- **ที่อยู่ตรรกะ** แสดงเป็นตำแหน่งสัมพัทธ์กับจุดเริ่มต้นของโปรแกรม คำสั่งในโปรแกรมมีเพียงที่อยู่ตรรกะเท่านั้น
- **ที่อยู่ทางกายภาพ** คือตำแหน่งจริงในหน่วยความจำหลัก เมื่อโปรเซสเซอร์ประมวลผลกระบวนการ มันจะแปลงจากที่อยู่ตรรกะเป็นที่อยู่ทางกายภาพโดยอัตโนมัติโดยการบวก **ที่อยู่ฐาน (base address)** ปัจจุบันของกระบวนการเข้ากับที่อยู่ตรรกะแต่ละที่อยู่

### การแบ่งหน้า (Paging)

พาร์ทิชันขนาดคงที่ที่ไม่เท่ากันและพาร์ทิชันขนาดแปรผันต่างก็ไม่มีประสิทธิภาพในการใช้หน่วยความจำ สมมติว่าหน่วยความจำถูกแบ่งออกเป็นส่วนที่มีขนาดเท่ากันและค่อนข้างเล็ก และแต่ละกระบวนการถูกแบ่งออกเป็นส่วนย่อยขนาดคงที่เล็ก ๆ ด้วย ส่วนย่อยของโปรแกรมเรียกว่า **หน้า (pages)** สามารถถูกกำหนดให้กับส่วนย่อยของหน่วยความจำที่ว่าง เรียกว่า **เฟรม (frames)** หรือเฟรมหน้า (page frames) อย่างมากก็มีเพียงเศษส่วนของหน้าสุดท้ายที่สูญเปล่า

OS รักษา **ตารางหน้า (page table)** สำหรับแต่ละกระบวนการ ตารางหน้าแสดงตำแหน่งเฟรมสำหรับแต่ละหน้าของกระบวนการ ภายในโปรแกรม ที่อยู่ตรรกะแต่ละที่อยู่ประกอบด้วย หมายเลขหน้า (page number) และที่อยู่สัมพัทธ์ภายในหน้า การแปลที่อยู่ตรรกะเป็นที่อยู่ทางกายภาพยังคงทำโดยฮาร์ดแวร์โปรเซสเซอร์ เมื่อได้รับที่อยู่ตรรกะ (หมายเลขหน้า, ที่อยู่สัมพัทธ์) โปรเซสเซอร์จะใช้ตารางหน้าเพื่อสร้างที่อยู่ทางกายภาพ (หมายเลขเฟรม, ที่อยู่สัมพัทธ์)

แนวทางนี้แก้ปัญหาที่เกิดขึ้นก่อนหน้านี้ หน่วยความจำหลักถูกแบ่งออกเป็นเฟรมขนาดเล็กที่เท่ากันจำนวนมาก แต่ละกระบวนการถูกแบ่งออกเป็นหน้าขนาดเฟรม กระบวนการขนาดเล็กต้องการหน้าน้อย กระบวนการขนาดใหญ่ต้องการหน้ามาก

### หน่วยความจำเสมือน (Virtual Memory)

#### การแบ่งหน้าตามต้องการ (Demand Paging)

**การแบ่งหน้าตามต้องการ (demand paging)** หมายถึงการที่แต่ละหน้าของกระบวนการจะถูกนำเข้ามาเฉพาะเมื่อต้องการเท่านั้น พิจารณากระบวนการขนาดใหญ่ที่ประกอบด้วยโปรแกรมยาวและอาร์เรย์ข้อมูลจำนวนหนึ่ง ในช่วงเวลาสั้น ๆ การประมวลผลอาจถูกจำกัดอยู่ในส่วนเล็ก ๆ ของโปรแกรม นี่คือ **หลักการเฉพาะถิ่น (principle of locality)**

เมื่อโปรแกรมแตกแขนงไปยังคำสั่งในหน้าที่ไม่ได้อยู่ในหน่วยความจำหลัก หรืออ้างอิงข้อมูลในหน้าที่ไม่ได้อยู่ในหน่วยความจำ จะเกิด **ข้อผิดพลาดหน้า (page fault)** ซึ่งบอก OS ให้นำหน้าที่ต้องการมา ดังนั้น ในช่วงเวลาใดก็ตาม มีเพียงไม่กี่หน้าของกระบวนการที่อยู่ในหน่วยความจำ ทำให้สามารถรักษากระบวนการจำนวนมากขึ้นในหน่วยความจำได้

เมื่อ OS นำหน้าหนึ่งเข้ามา มันต้องส่งหน้าอื่นออกไป นี่เรียกว่า **การแทนที่หน้า (page replacement)** หากส่งหน้าออกไปก่อนที่หน้านั้นจะถูกใช้งาน ก็ต้องนำกลับมาใหม่เกือบทันที การทำเช่นนี้มากเกินไปนำไปสู่ภาวะที่เรียกว่า **การล้น (thrashing):** โปรเซสเซอร์ใช้เวลาส่วนใหญ่ในการสลับหน้าแทนที่จะประมวลผลคำสั่ง เทคนิคที่อาจมีประสิทธิผลคือ **ที่ใช้น้อยที่สุดล่าสุด (Least Recently Used — LRU)**

ด้วยการแบ่งหน้าตามต้องการ ไม่จำเป็นต้องโหลดกระบวนการทั้งหมดเข้าหน่วยความจำหลัก ซึ่งมีผลที่น่าทึ่ง: **กระบวนการสามารถมีขนาดใหญ่กว่าหน่วยความจำหลักทั้งหมดได้** เนื่องจากกระบวนการทำงานในหน่วยความจำหลักเท่านั้น หน่วยความจำนั้นจึงเรียกว่า **หน่วยความจำจริง (real memory)** แต่โปรแกรมเมอร์หรือผู้ใช้รับรู้หน่วยความจำที่ใหญ่กว่ามาก ซึ่งจัดสรรบนดิสก์ หน่วยความจำนี้จึงเรียกว่า **หน่วยความจำเสมือน (virtual memory)**

#### โครงสร้างตารางหน้า (Page Table Structure)

กลไกพื้นฐานสำหรับการอ่านคำจากหน่วยความจำเกี่ยวข้องกับการแปลที่อยู่เสมือน (virtual address) ซึ่งประกอบด้วยหมายเลขหน้าและออฟเซต ไปเป็นที่อยู่ทางกายภาพ ประกอบด้วยหมายเลขเฟรมและออฟเซต โดยใช้ตารางหน้า

ระบบส่วนใหญ่มีตารางหน้าหนึ่งตารางต่อกระบวนการ แต่แต่ละกระบวนการสามารถครองหน่วยความจำเสมือนจำนวนมาก ตัวอย่างเช่น ในสถาปัตยกรรม VAX แต่ละกระบวนการสามารถมีหน่วยความจำเสมือนได้ถึง 2 กิกะไบต์

เพื่อแก้ปัญหานี้ ระบบหน่วยความจำเสมือนส่วนใหญ่จัดเก็บตารางหน้าในหน่วยความจำเสมือนแทนหน่วยความจำจริง โปรเซสเซอร์บางรุ่นใช้ **โครงสร้างสองระดับ (two-level scheme)** เพื่อจัดระเบียบตารางหน้าขนาดใหญ่ โดยมี **ไดเรกทอรีหน้า (page directory)** ที่แต่ละรายการชี้ไปยังตารางหน้า

อีกแนวทางหนึ่งคือการใช้ **โครงสร้างตารางหน้าผกผัน (inverted page table)** ที่ใช้ในสถาปัตยกรรม PowerPC, UltraSPARC และ IA-64 ในแนวทางนี้ หมายเลขหน้าของที่อยู่เสมือนถูกแมปเป็นค่าแฮช (hash value) ซึ่งเป็นตัวชี้ไปยังตารางหน้าผกผัน มีรายการหนึ่งสำหรับแต่ละเฟรมหน้าหน่วยความจำจริง แทนที่จะเป็นหนึ่งต่อหน้าเสมือน

### บัฟเฟอร์ค้นหาการแปล (Translation Lookaside Buffer)

การอ้างอิงหน่วยความจำเสมือนทุกครั้งอาจทำให้เกิดการเข้าถึงหน่วยความจำทางกายภาพสองครั้ง ได้แก่ หนึ่งครั้งเพื่อดึงรายการตารางหน้าที่เหมาะสม และหนึ่งครั้งเพื่อดึงข้อมูลที่ต้องการ เพื่อแก้ปัญหานี้ ระบบหน่วยความจำเสมือนส่วนใหญ่ใช้แคชพิเศษสำหรับรายการตารางหน้า ซึ่งมักเรียกว่า **บัฟเฟอร์ค้นหาการแปล (Translation Lookaside Buffer — TLB)**

TLB ทำงานในลักษณะเดียวกับแคชหน่วยความจำและบรรจุรายการตารางหน้าที่ถูกใช้บ่อยที่สุดเมื่อเร็ว ๆ นี้ ตามหลักการเฉพาะถิ่น การอ้างอิงหน่วยความจำเสมือนส่วนใหญ่จะอยู่ในตำแหน่งในหน้าที่ใช้ล่าสุด ดังนั้น การอ้างอิงส่วนใหญ่จะเกี่ยวข้องกับรายการตารางหน้าในแคช

ที่อยู่เสมือนโดยทั่วไปจะอยู่ในรูปแบบหมายเลขหน้า ออฟเซต ก่อนอื่น ระบบหน่วยความจำจะตรวจสอบ TLB เพื่อดูว่ารายการตารางหน้าที่ตรงกันนั้นมีอยู่หรือไม่ หากมี ที่อยู่จริง (ทางกายภาพ) จะถูกสร้างโดยการรวมหมายเลขเฟรมกับออฟเซต หากไม่มี รายการจะถูกเข้าถึงจากตารางหน้าในหน่วยความจำหลัก เมื่อสร้างที่อยู่จริงได้แล้ว แคชจะถูกตรวจสอบเพื่อดูว่าบล็อกที่มีคำนั้นมีอยู่หรือไม่

### การแบ่งเซกเมนต์ (Segmentation)

มีอีกวิธีหนึ่งที่หน่วยความจำที่ระบุได้สามารถแบ่งย่อยได้ เรียกว่า **การแบ่งเซกเมนต์ (segmentation)** ในขณะที่การแบ่งหน้าไม่ปรากฏแก่โปรแกรมเมอร์และมีวัตถุประสงค์เพื่อให้โปรแกรมเมอร์มีพื้นที่แอดเดรสที่ใหญ่กว่า การแบ่งเซกเมนต์มักปรากฏแก่โปรแกรมเมอร์และให้ความสะดวกในการจัดระเบียบโปรแกรมและข้อมูล

การแบ่งเซกเมนต์ให้โปรแกรมเมอร์มองหน่วยความจำว่าประกอบด้วยพื้นที่แอดเดรสหรือเซกเมนต์หลายส่วน เซกเมนต์มีขนาดแปรผัน โดยทั่วไปโปรแกรมเมอร์หรือ OS จะกำหนดโปรแกรมและข้อมูลให้กับเซกเมนต์ต่าง ๆ การอ้างอิงหน่วยความจำประกอบด้วยที่อยู่ในรูปแบบ (หมายเลขเซกเมนต์, ออฟเซต)

การแบ่งเซกเมนต์มีข้อได้เปรียบหลายประการสำหรับโปรแกรมเมอร์เมื่อเปรียบเทียบกับพื้นที่แอดเดรสที่ไม่ได้แบ่งเซกเมนต์:

1. **ง่ายต่อการจัดการโครงสร้างข้อมูลที่เติบโต:** หากโปรแกรมเมอร์ไม่รู้ล่วงหน้าว่าโครงสร้างข้อมูลขนาดใดจะเป็น ก็ไม่จำเป็นต้องเดา โครงสร้างข้อมูลสามารถถูกกำหนดให้กับเซกเมนต์ของตัวเอง และ OS จะขยายหรือลดขนาดเซกเมนต์ตามที่จำเป็น
2. **ให้โปรแกรมถูกเปลี่ยนแปลงและคอมไพล์ใหม่อย่างเป็นอิสระ:** โดยไม่จำเป็นต้องเชื่อมโยงและโหลดโปรแกรมทั้งชุดใหม่
3. **เหมาะสำหรับการแบ่งปันระหว่างกระบวนการ:** โปรแกรมเมอร์สามารถวางโปรแกรมยูทิลิตีหรือตารางข้อมูลที่มีประโยชน์ในเซกเมนต์ที่กระบวนการอื่นสามารถระบุได้
4. **เหมาะสำหรับการป้องกัน:** เนื่องจากเซกเมนต์สามารถสร้างขึ้นเพื่อบรรจุชุดโปรแกรมหรือข้อมูลที่กำหนดไว้อย่างชัดเจน โปรแกรมเมอร์หรือผู้ดูแลระบบสามารถกำหนดสิทธิ์การเข้าถึงได้อย่างสะดวก

## 8.4 การจัดการหน่วยความจำของ Intel x86

ตั้งแต่มีการเปิดตัวสถาปัตยกรรม 32 บิต ไมโครโปรเซสเซอร์ได้พัฒนาแผนการจัดการหน่วยความจำที่ซับซ้อนซึ่งสร้างต่อยอดจากบทเรียนที่ได้รับจากระบบขนาดกลางและขนาดใหญ่

### พื้นที่แอดเดรส (Address Spaces)

x86 มีฮาร์ดแวร์สำหรับทั้งการแบ่งเซกเมนต์และการแบ่งหน้า ทั้งสองกลไกสามารถปิดใช้งานได้ ซึ่งให้ผู้ใช้เลือกจากมุมมองของหน่วยความจำสี่แบบที่แตกต่างกัน:

- **หน่วยความจำที่ไม่แบ่งเซกเมนต์และไม่แบ่งหน้า (Unsegmented unpaged memory):** ในกรณีนี้ ที่อยู่เสมือนเหมือนกับที่อยู่ทางกายภาพ มีประโยชน์สำหรับโปรแกรมประยุกต์ควบคุมที่มีความซับซ้อนต่ำและต้องการประสิทธิภาพสูง
- **หน่วยความจำที่ไม่แบ่งเซกเมนต์แต่แบ่งหน้า (Unsegmented paged memory):** หน่วยความจำมองว่าเป็นพื้นที่แอดเดรสเชิงเส้นที่แบ่งหน้า การป้องกันและการจัดการหน่วยความจำทำผ่านการแบ่งหน้า บางระบบปฏิบัติการ เช่น Berkeley UNIX นิยมใช้แบบนี้
- **หน่วยความจำที่แบ่งเซกเมนต์แต่ไม่แบ่งหน้า (Segmented unpaged memory):** หน่วยความจำมองว่าเป็นชุดของพื้นที่แอดเดรสตรรกะ ข้อได้เปรียบของมุมมองนี้เหนือแนวทางการแบ่งหน้าคือให้การป้องกันถึงระดับไบต์เดียวหากจำเป็น นอกจากนี้ต่างจากการแบ่งหน้า มันรับประกันว่าตารางการแปลที่จำเป็น (segment table) จะอยู่บนชิปเมื่อเซกเมนต์อยู่ในหน่วยความจำ
- **หน่วยความจำที่แบ่งเซกเมนต์และแบ่งหน้า (Segmented paged memory):** การแบ่งเซกเมนต์ใช้เพื่อกำหนดพาร์ทิชันหน่วยความจำตรรกะที่อยู่ภายใต้การควบคุมการเข้าถึง และการแบ่งหน้าใช้เพื่อจัดการการจัดสรรหน่วยความจำภายในพาร์ทิชัน ระบบปฏิบัติการเช่น UNIX System V นิยมใช้แบบนี้

### การแบ่งเซกเมนต์ใน x86

เมื่อใช้การแบ่งเซกเมนต์ ที่อยู่เสมือนแต่ละที่อยู่ (เรียกว่าที่อยู่ตรรกะในเอกสาร Intel) ประกอบด้วย การอ้างอิงเซกเมนต์ 16 บิตและออฟเซต 32 บิต สองบิตของการอ้างอิงเซกเมนต์เกี่ยวข้องกับกลไกการป้องกัน เหลือ 14 บิตสำหรับระบุเซกเมนต์เฉพาะ

ดังนั้น ด้วยหน่วยความจำที่ไม่แบ่งเซกเมนต์ หน่วยความจำเสมือนของผู้ใช้คือ 2^32 = 4 กิกะไบต์ ด้วยหน่วยความจำที่แบ่งเซกเมนต์ พื้นที่หน่วยความจำเสมือนทั้งหมดที่ผู้ใช้มองเห็นคือ 2^46 = 64 เทระไบต์ พื้นที่แอดเดรสทางกายภาพใช้แอดเดรส 32 บิตสำหรับสูงสุด 4 กิกะไบต์

มีการป้องกันสองรูปแบบที่เกี่ยวข้องกับแต่ละเซกเมนต์ ได้แก่ ระดับสิทธิพิเศษ (privilege level) และแอตทริบิวต์การเข้าถึง มีระดับสิทธิพิเศษสี่ระดับ จากที่ได้รับการป้องกันมากที่สุด (ระดับ 0) ไปจนถึงน้อยที่สุดที่ได้รับการป้องกัน (ระดับ 3)

แอตทริบิวต์การเข้าถึงของเซกเมนต์ข้อมูลระบุว่าอนุญาตให้อ่าน/เขียนหรืออ่านอย่างเดียว สำหรับเซกเมนต์โปรแกรม แอตทริบิวต์การเข้าถึงระบุการเข้าถึงแบบอ่าน/ประมวลผลหรืออ่านอย่างเดียว

กลไกการแปลแอดเดรสสำหรับการแบ่งเซกเมนต์เกี่ยวข้องกับการแมปที่อยู่เสมือนเป็นที่อยู่เชิงเส้น (linear address) ตัวเลือกเซกเมนต์ (segment selector) ประกอบด้วย:

- **ตัวบ่งชี้ตาราง (Table Indicator — TI):** ระบุว่าควรใช้ตารางเซกเมนต์ส่วนกลาง (global segment table) หรือตารางเซกเมนต์ท้องถิ่น (local segment table) สำหรับการแปล
- **หมายเลขเซกเมนต์ (Segment Number):** หมายเลขของเซกเมนต์ ซึ่งทำหน้าที่เป็นดัชนีเข้าตารางเซกเมนต์
- **ระดับสิทธิพิเศษที่ร้องขอ (Requested Privilege Level — RPL):** ระดับสิทธิพิเศษที่ร้องขอสำหรับการเข้าถึงนี้

แต่ละรายการในตารางเซกเมนต์ประกอบด้วย 64 บิต โดยมีฟิลด์ต่าง ๆ ดังนี้:

| พารามิเตอร์ | คำอธิบาย |
|---|---|
| ฐาน (Base) | กำหนดที่อยู่เริ่มต้นของเซกเมนต์ภายในพื้นที่แอดเดรสเชิงเส้น 4 กิกะไบต์ |
| บิต D/B | ในเซกเมนต์โค้ด นี่คือบิต D และระบุว่าตัวดำเนินการและโหมดแอดเดรสเป็น 16 หรือ 32 บิต |
| ระดับสิทธิพิเศษ Descriptor (DPL) | ระบุระดับสิทธิพิเศษของเซกเมนต์ที่อ้างถึงโดย descriptor นี้ |
| บิตความละเอียด (Granularity — G) | ระบุว่าฟิลด์ขีดจำกัดจะถูกตีความในหน่วยหนึ่งไบต์หรือ 4 กิโลไบต์ |
| ขีดจำกัด (Limit) | กำหนดขนาดของเซกเมนต์ |
| บิต S | กำหนดว่าเซกเมนต์ที่กำหนดเป็น system segment หรือ code/data segment |
| บิตเซกเมนต์ปรากฏ (Segment Present — P) | ใช้สำหรับระบบที่ไม่แบ่งหน้า ระบุว่าเซกเมนต์อยู่ในหน่วยความจำหลักหรือไม่ |
| ประเภท (Type) | แยกความแตกต่างระหว่างเซกเมนต์ประเภทต่าง ๆ และระบุแอตทริบิวต์การเข้าถึง |

### การแบ่งหน้าใน x86

กลไกการแบ่งหน้าของ x86 เป็นการดำเนินการค้นหาตารางสองระดับ ระดับแรกคือ **ไดเรกทอรีหน้า (page directory)** ซึ่งบรรจุได้สูงสุด 1,024 รายการ นี่แบ่งพื้นที่หน่วยความจำเชิงเส้น 4 กิกะไบต์ออกเป็น 1,024 กลุ่มหน้า แต่ละกลุ่มมีตารางหน้าของตัวเอง และแต่ละกลุ่มมีขนาด 4 เมกะไบต์ ตารางหน้าแต่ละตารางบรรจุได้สูงสุด 1,024 รายการ โดยแต่ละรายการสอดคล้องกับหน้า 4 กิโลไบต์เดียว

x86 ยังใช้ translation lookaside buffer ซึ่งสามารถเก็บรายการตารางหน้าได้ 32 รายการ ทุกครั้งที่ไดเรกทอรีหน้าเปลี่ยนแปลง บัฟเฟอร์จะถูกล้าง

สุดท้าย x86 มีส่วนขยายใหม่คือการจัดเตรียมขนาดหน้าสองขนาด หากบิต PSE (page size extension) ถูกตั้งเป็น 1 หน่วยการแบ่งหน้าจะอนุญาตให้โปรแกรมเมอร์ OS กำหนดหน้าเป็นขนาด 4 กิโลไบต์หรือ 4 เมกะไบต์

## 8.5 การจัดการหน่วยความจำของ ARM

ARM ให้สถาปัตยกรรมระบบหน่วยความจำเสมือนที่หลากหลายที่สามารถปรับให้เหมาะกับความต้องการของนักออกแบบระบบฝังตัว (embedded system designer)

### การจัดระบบหน่วยความจำ (Memory System Organization)

ฮาร์ดแวร์แปลหน่วยความจำเสมือน (virtual memory translation hardware) ใช้ตารางหนึ่งหรือสองระดับสำหรับการแปลจากที่อยู่เสมือนเป็นที่อยู่ทางกายภาพ TLB เป็นแคชของรายการตารางหน้าล่าสุด หากรายการมีอยู่ใน TLB TLB จะส่งที่อยู่ทางกายภาพไปยังหน่วยความจำหลักโดยตรงสำหรับการอ่านหรือเขียน

รายการในตารางการแปลยังรวมบิตควบคุมการเข้าถึง (access control bits) ซึ่งกำหนดว่ากระบวนการที่กำหนดสามารถเข้าถึงส่วนที่กำหนดของหน่วยความจำได้หรือไม่ หากการเข้าถึงถูกปฏิเสธ ฮาร์ดแวร์ควบคุมการเข้าถึงจะส่งสัญญาณยกเลิก (abort signal) ไปยังโปรเซสเซอร์ ARM

### การแปลแอดเดรสหน่วยความจำเสมือน (Virtual Memory Address Translation)

ARM รองรับการเข้าถึงหน่วยความจำตามส่วน (sections) หรือหน้า (pages):

- **ซูเปอร์เซกชัน (Supersections — ไม่บังคับ):** ประกอบด้วยบล็อก 16 เมกะไบต์ของหน่วยความจำหลัก
- **เซกชัน (Sections):** ประกอบด้วยบล็อก 1 เมกะไบต์ของหน่วยความจำหลัก
- **หน้าขนาดใหญ่ (Large pages):** ประกอบด้วยบล็อก 64 กิโลไบต์ของหน่วยความจำหลัก
- **หน้าขนาดเล็ก (Small pages):** ประกอบด้วยบล็อก 4 กิโลไบต์ของหน่วยความจำหลัก

เซกชันและซูเปอร์เซกชันรองรับเพื่อให้แมปพื้นที่หน่วยความจำขนาดใหญ่ในขณะที่ใช้เพียงรายการเดียวใน TLB ตารางการแปลที่เก็บในหน่วยความจำหลักมีสองระดับ:

- **ตาราง Level 1:** เก็บ descriptor ระดับ 1 ที่มีที่อยู่ฐานและคุณสมบัติการแปลสำหรับ Section และ Supersection และคุณสมบัติการแปลและตัวชี้ไปยังตาราง level 2 สำหรับหน้าขนาดใหญ่หรือขนาดเล็ก
- **ตาราง Level 2:** เก็บ descriptor ระดับ 2 ที่มีที่อยู่ฐานและคุณสมบัติการแปลสำหรับหน้าขนาดเล็กหรือขนาดใหญ่ ตาราง level 2 ต้องใช้หน่วยความจำ 1 กิโลไบต์

หน่วยจัดการหน่วยความจำ (Memory-Management Unit — MMU) แปลที่อยู่เสมือนที่สร้างโดยโปรเซสเซอร์ไปเป็นที่อยู่ทางกายภาพ และตรวจสอบสิทธิ์การเข้าถึง

สำหรับหน้าขนาดเล็ก การแปลที่อยู่เสมือนทำงานดังนี้ มีตาราง L1 เดียวที่มี 4K รายการ 32 บิต แต่ละรายการ L1 ชี้ไปยังตาราง L2 ที่มี 256 รายการ 32 บิต แต่ละรายการ L2 ชี้ไปยังหน้า 4 กิโลไบต์ในหน่วยความจำหลัก ที่อยู่เสมือน 32 บิตถูกตีความดังนี้: 12 บิตนัยสำคัญที่สุดเป็นดัชนีเข้าตาราง L1 อีก 8 บิตถัดไปเป็นดัชนีเข้าตาราง L2 ที่เกี่ยวข้อง และ 12 บิตนัยสำคัญน้อยที่สุดเป็นดัชนีไบต์ในหน้าที่เกี่ยวข้องในหน่วยความจำหลัก

### รูปแบบการจัดการหน่วยความจำ (Memory-Management Formats)

สำหรับตาราง L1 แต่ละรายการเป็น descriptor ของวิธีที่ช่วงที่อยู่เสมือน 1 เมกะไบต์ที่เกี่ยวข้องถูกแมป แต่ละรายการมีรูปแบบทางเลือกสี่แบบ:

- **บิต [1:0] = 00:** ที่อยู่เสมือนที่เกี่ยวข้องไม่ถูกแมป และการพยายามเข้าถึงจะสร้างข้อผิดพลาดการแปล (translation fault)
- **บิต [1:0] = 01:** รายการให้ที่อยู่ทางกายภาพของตาราง L2 ซึ่งระบุวิธีที่ช่วงที่อยู่เสมือนที่เกี่ยวข้องถูกแมป
- **บิต [1:0] = 01 และบิต 19 = 0:** รายการเป็น descriptor ของ section สำหรับที่อยู่เสมือนที่เกี่ยวข้อง
- **บิต [1:0] = 01 และบิต 19 = 1:** รายการเป็น descriptor ของ supersection สำหรับที่อยู่เสมือนที่เกี่ยวข้อง

### การควบคุมการเข้าถึง (Access Control)

บิตควบคุมการเข้าถึง AP ในแต่ละรายการตารางควบคุมการเข้าถึงพื้นที่หน่วยความจำโดยกระบวนการที่กำหนด พื้นที่หน่วยความจำสามารถถูกกำหนดเป็น ไม่มีการเข้าถึง อ่านอย่างเดียว หรืออ่าน-เขียน นอกจากนี้ พื้นที่สามารถถูกกำหนดเป็นการเข้าถึงสิทธิพิเศษเท่านั้น สงวนไว้สำหรับใช้โดย OS และไม่ใช่โดยโปรแกรมประยุกต์

ARM ยังใช้แนวคิดของ **โดเมน (domain)** ซึ่งเป็นกลุ่มของ section และ/หรือหน้าที่มีสิทธิ์การเข้าถึงเฉพาะ สถาปัตยกรรม ARM รองรับ 16 โดเมน คุณลักษณะโดเมนให้กระบวนการหลายกระบวนการใช้ตารางการแปลเดียวกันในขณะที่ยังรักษาการป้องกันซึ่งกันและกัน

แต่ละรายการตารางหน้าและรายการ TLB มีฟิลด์ที่ระบุว่ารายการนั้นอยู่ในโดเมนใด ฟิลด์ 2 บิตใน Domain Access Control Register ควบคุมการเข้าถึงแต่ละโดเมน มีการรองรับการเข้าถึงโดเมนสองประเภท:

- **ลูกค้า (Clients):** ผู้ใช้โดเมน (ประมวลผลโปรแกรมและเข้าถึงข้อมูล) ที่ต้องปฏิบัติตามสิทธิ์การเข้าถึงของ section และ/หรือหน้าแต่ละหน่วยที่ประกอบเป็นโดเมนนั้น
- **ผู้จัดการ (Managers):** ควบคุมพฤติกรรมของโดเมน (section และหน้าปัจจุบันในโดเมน และการเข้าถึงโดเมน) และข้ามผ่านสิทธิ์การเข้าถึงสำหรับรายการตารางในโดเมนนั้น

โปรแกรมหนึ่งสามารถเป็นลูกค้าของโดเมนบางส่วน ผู้จัดการของโดเมนบางส่วน และไม่มีสิทธิ์เข้าถึงโดเมนที่เหลือ ซึ่งให้การป้องกันหน่วยความจำที่ยืดหยุ่นมากสำหรับโปรแกรมที่เข้าถึงทรัพยากรหน่วยความจำต่างกัน

พารามิเตอร์การจัดการหน่วยความจำของ ARM:

| พารามิเตอร์ | คำอธิบาย |
|---|---|
| สิทธิ์การเข้าถึง AP, APX | บิตเหล่านี้ควบคุมการเข้าถึงพื้นที่หน่วยความจำที่สอดคล้องกัน หากทำการเข้าถึงพื้นที่หน่วยความจำโดยไม่มีสิทธิ์ที่จำเป็น จะเกิด Permission Fault |
| บิต Bufferable (B) | กำหนด ร่วมกับบิต TEX วิธีใช้บัฟเฟอร์การเขียนสำหรับหน่วยความจำ cacheable |
| บิต Cacheable (C) | กำหนดว่าพื้นที่หน่วยความจำนี้สามารถแมปผ่านแคชได้หรือไม่ |
| โดเมน (Domain) | กลุ่มของพื้นที่หน่วยความจำ สามารถใช้การควบคุมการเข้าถึงตามโดเมน |
| not Global (nG) | กำหนดว่าการแปลควรถูกทำเครื่องหมายเป็นส่วนกลาง (0) หรือเฉพาะกระบวนการ (1) |
| Shared (S) | กำหนดว่าการแปลเป็นสำหรับหน่วยความจำที่ไม่แบ่งปัน (0) หรือแบ่งปัน (1) |
| SBZ | ควรเป็นศูนย์ (Should be zero) |
| Type Extension (TEX) | บิตเหล่านี้ร่วมกับบิต B และ C ควบคุมการเข้าถึงแคช วิธีใช้บัฟเฟอร์การเขียน และว่าพื้นที่หน่วยความจำสามารถแบ่งปันได้ |
| Execute Never (XN) | กำหนดว่าพื้นที่ประมวลผลได้ (0) หรือไม่ประมวลผลได้ (1) |

### ปัญหา

**8.1** สมมติว่ามีคอมพิวเตอร์ที่ทำงานหลายโปรแกรมซึ่งแต่ละงานมีลักษณะเหมือนกัน ในช่วงการคำนวณหนึ่งช่วง T สำหรับงาน ครึ่งหนึ่งของเวลาใช้ใน I/O และอีกครึ่งหนึ่งในกิจกรรมโปรเซสเซอร์ แต่ละงานทำงานรวมทั้งหมด N ช่วง สมมติว่าใช้ลำดับความสำคัญแบบวนรอบอย่างง่าย และการดำเนินการ I/O สามารถทับซ้อนกับการดำเนินการโปรเซสเซอร์ได้ กำหนดปริมาณต่อไปนี้:
- เวลาหมุนเวียน = เวลาจริงในการทำงานหนึ่งงานให้เสร็จ
- ปริมาณงาน = จำนวนงานเฉลี่ยที่เสร็จต่อช่วงเวลา T
- การใช้งานโปรเซสเซอร์ = เปอร์เซ็นต์ของเวลาที่โปรเซสเซอร์ทำงานอยู่ (ไม่รอ)

**8.2** โปรแกรมที่ผูกกับ I/O คือโปรแกรมที่หากรันโดยลำพัง จะใช้เวลารอ I/O มากกว่าใช้โปรเซสเซอร์ โปรแกรมที่ผูกกับโปรเซสเซอร์เป็นตรงกันข้าม สมมติว่าอัลกอริทึมการจัดตารางเวลาระยะสั้นเลือกโปรแกรมที่ใช้เวลาโปรเซสเซอร์น้อยในช่วงเวลาที่ผ่านมา จงอธิบายว่าทำไมอัลกอริทึมนี้จึงเลือกโปรแกรมที่ผูกกับ I/O แต่ก็ไม่ปฏิเสธเวลาโปรเซสเซอร์แก่โปรแกรมที่ผูกกับโปรเซสเซอร์อย่างถาวร
